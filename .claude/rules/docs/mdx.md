# MDX Documentation Rules

The docs site is Next.js + Fumadocs, static-exported to https://eventum.run. Technical content lives in `../docs/content/docs/` (Fumadocs MDX collections); top-level pages in `../docs/app/` (Next.js routes). Blog posts have their own rules described in `blog.md`.

## Frontmatter

Required: `title`, `description`. Optional: `icon` (Lucide icon name).

```yaml
---
title: Page Title
description: One-line SEO description shown on link cards and search.
icon: BookA
---
```

## Writing style

- **End-user language, not engine jargon.** Describe what the user sees and configures, not internal terms (`Plugin.produce()`, `EventContext`, `FSM picker`).
- **Em dash (`—`) is allowed** in MDX prose; source code elsewhere uses single `-` only.

## Parameter tables

5 columns: `Name | Type | Default | Constraints | Description`. Use `—` in `Default` / `Constraints` when not applicable.

## Code blocks

Language tag required; `title="..."` attribute optional for a filename label:

```` markdown
```yaml title="generator.yml"
input:
  - cron:
      expression: "* * * * * *"
```
````

## Fumadocs components

Prefer these over raw HTML:

- `<Card href="..." title="..." description="..." />` - link card; wrap multiple in `<Cards>` for a grid.
- `<Callout type="info|warn|error">` - info / warning / error box.
- `<Steps>` with `<Step>` children - numbered procedures.
- `<Accordions>` with `<Accordion title="...">` - collapsible sections.
- `<Tabs items={["A", "B"]}>` with `<Tab>` children - tabbed variants.
- `<ThemedImage lightSrc="..." darkSrc="..." alt="..." />` - light / dark screenshots.
- `<Files>` - file tree.
- `<Mermaid chart={`...`} />` - diagrams.
- `<APIPage>` - only in auto-generated API pages; don't use manually.

## Navigation (`meta.json`)

Pages omitted from the `pages` array are hidden - always add new pages explicitly.

- `"index"` - regular page reference.
- `"..."` - include remaining files alphabetically.
- `"...prefix"` - include files starting with `prefix`.
- `"---Section---"` - section divider.
- `"[Icon][Label](/path)"` - custom entry with a Lucide icon.

## Auto-generated files (never edit)

- `docs/.source/` - generated by `fumadocs-mdx` on `postinstall`.
- `docs/content/docs/api/` - generated from `docs/public/schemas/eventum-openapi.json` by `scripts/generate-docs.ts`. Regenerate via `pnpm generate-api-docs`.

## Build verification

Run `pnpm build` after content changes - catches broken MDX, missing nav entries, bad links, type errors.

## Changelog entries

One file per release at `../docs/content/docs/changelog/<version>.mdx`. `meta.json` uses `{"pages": ["z...a"]}` for reverse-alpha order (newest on top).

Frontmatter `title` carries a release-type emoji:

- `🎉 X.0.0` - major release.
- `🚩 X.Y.0` - minor release.
- `🛠️ X.Y.Z` - patch release.

Body opens with `Released **Month DD, YYYY**` on its own line, optionally followed by a one-paragraph summary.

H2 sections use emoji prefixes - include only what applies:

- `## 🚀 New features`
- `## 🐛 Bug fixes`
- `## ⚡ Performance`
- `## 🏗️ Architecture`
- `## 🧪 Testing`
- `## 📝 Other changes`

Use `### Subsection` inside a section when item count warrants grouping (e.g. `### Input plugins`, `### Output plugins` inside `🚀 New features`).

Write entries from the user's perspective - what they can do, not what the code does.

## Links

- Internal: relative path to `.mdx` without extension - Fumadocs resolves it.
- External: full URL - Fumadocs opens in a new tab automatically.
- API reference: paths under `/docs/api/...`.
